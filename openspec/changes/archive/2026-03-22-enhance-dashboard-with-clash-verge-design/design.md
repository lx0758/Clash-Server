## Context

当前项目的仪表盘功能过于简单，仅包含 4 个基础统计卡片，缺少现代代理客户端（如 Clash Verge）常见的丰富功能。后端 WebSocket 已提供完整的实时流量数据推送（traffic, connections, memory, core_status, logs），代理 API 也支持模式切换、延迟测试等操作，但前端未能充分利用这些数据提供良好的用户体验。

现有的响应式系统（useBreakpoint）已建立完善的断点机制（xs/sm/md/lg/xl），但 Dashboard 页面未充分利用响应式布局能力。移动端用户无法在导航栏快速查看流量状态，需要跳转到仪表盘页面才能查看实时数据。

技术栈：
- 前端：Vue 3 + TypeScript + Element Plus + Pinia + Vue Router
- 实时数据：WebSocket (已在 useWebSocket composable 中封装)
- 响应式：自定义 useBreakpoint composable (已实现 xs/sm/md/lg/xl 断点)

## Goals / Non-Goals

**Goals:**

1. 提供丰富的仪表盘数据可视化，包括实时流量图表和 6 个统计卡片
2. 支持代理模式快速切换，提供直观的交互体验
3. 显示当前代理和网络设置状态，方便用户快速控制
4. 显示订阅配置信息，包括流量使用和过期时间
5. 在移动端导航栏添加流量指示器，实时显示上行/下行速度
6. 所有组件完全响应式，适配移动端（<768px）、平板端（768-1200px）、桌面端（>1200px）
7. 使用原生 Canvas API 绘制流量图表，避免引入额外依赖

**Non-Goals:**

1. 不修改后端 API 和 WebSocket 数据格式（现有能力已满足需求）
2. 不改变现有的认证和权限系统
3. 不引入重型第三方图表库（如 ECharts、Chart.js），保持轻量级
4. 不实现完整的用户个性化布局配置（如 Clash Verge 的卡片自定义）
5. 不实现速度测试和 IP 信息等高级功能（留待未来增强）

## Decisions

### 1. 使用原生 Canvas API 绘制流量图表

**决策：** 不引入第三方图表库，使用原生 Canvas API 实现 TrafficGraph 组件。

**理由：**
- 需求相对简单：实时双曲线图（上行/下行），无需复杂的图表功能
- 避免增加项目依赖和打包体积
- 性能可控：requestAnimationFrame + 手动数据缓冲区管理
- 学习成本和长期维护成本低

**替代方案考虑：**
- ECharts：功能强大但包体积大（~500KB gzipped），过度设计
- Chart.js：较轻量（~70KB gzipped）但仍增加依赖，定制化程度有限
- D3.js：学习曲线陡峭，适合复杂可视化，不适合实时简单图表

### 2. 移动端显示精简版流量图表

**决策：** 移动端（xs/sm 断点）显示高度 60-80px 的精简版流量图表，而不是隐藏或完全展开。

**理由：**
- 流量趋势是核心信息，移动端用户也需要快速查看
- 精简版占用空间小，不影响页面滚动
- 提供一致性体验：所有屏幕都有流量图表，只是细节程度不同

**替代方案考虑：**
- 完全隐藏：用户体验差，无法快速了解流量趋势
- 可折叠（默认隐藏）：增加交互复杂度，可能被忽视
- 完整显示：占用过多空间，影响移动端体验

### 3. 6 个统计卡片在移动端合并为 4 个

**决策：** 移动端合并"总上传量"和"总下载量"为"总流量"单卡片，减少移动端卡片数量。

**理由：**
- 移动端屏幕空间有限，减少卡片数提升可读性
- 总上传和总下载的信息密度相似，合并后仍保留核心信息
- 保持上行/下行速度的独立显示（实时性更重要）

**替代方案考虑：**
- 保持 6 个卡片：信息完整但布局拥挤
- 隐藏总上传/总下载：丢失重要历史数据
- 使用 Tab 切换：增加交互复杂度

### 4. 导航栏流量指示器仅在移动端显示

**决策：** TrafficIndicator 组件仅集成到移动端 Header（el-header v-if="isMobile"），桌面端不显示。

**理由：**
- 桌面端有完整的仪表盘页面，无需在导航栏重复显示
- 移动端 Header 空间有限，但流量信息对移动用户更重要
- 减少桌面端界面的视觉噪音

**替代方案考虑：**
- 所有屏幕都显示：桌面端冗余，增加视觉复杂度
- 侧边栏显示：桌面端侧边栏可折叠，流量信息可能被隐藏
- 弹出式显示：需要交互才能查看，不够直观

### 5. 使用 Element Plus Grid 系统实现响应式布局

**决策：** 使用 Element Plus 的 el-row 和 el-col 组件，配合现有的 useBreakpoint composable。

**理由：**
- 项目已使用 Element Plus，保持技术栈一致性
- el-col 的 :xs/:sm/:md/:lg/:xl 属性与 useBreakpoint 的断点一致
- 响应式声明式，代码清晰易维护

**替代方案考虑：**
- CSS Grid：功能强大但与 Element Plus 风格不统一
- Tailwind CSS：需要引入新依赖，增加项目复杂度
- 手动媒体查询：代码冗余，难以维护

### 6. 流量数据使用 60 秒缓冲区

**决策：** TrafficGraph 组件维护一个 60 个数据点的缓冲区（每秒推送一次数据），显示最近 60 秒的流量趋势。

**理由：**
- 60 秒足够展示流量变化趋势，同时保持性能
- 每秒推送频率与 WebSocket 实时性匹配
- 固定窗口大小，避免内存无限增长

**替代方案考虑：**
- 30 秒：窗口过短，趋势不明显
- 120 秒：内存占用增加，且长历史对实时监控价值有限
- 动态窗口：增加复杂度，性能不可控

## Risks / Trade-offs

### [Risk] 移动端性能问题（Canvas 绘制和大量 DOM 节点）

**缓解措施：**
- 使用 requestAnimationFrame 优化 Canvas 绘制频率（与屏幕刷新率同步）
- 移动端减少数据点数量（使用降采样或简化贝塞尔曲线）
- 使用 Vue 的 computed 和 onMounted/onUnmounted 生命周期优化组件卸载
- 移动端精简版图表减少绘制复杂度（去除网格线、图例等）

### [Risk] WebSocket 断开重连导致数据丢失

**缓解措施：**
- useWebSocket composable 已实现自动重连机制（3 秒延迟）
- 流量图表在重连后清空缓冲区，避免数据错乱
- 显示连接状态指示器（成功/断开/重连中）

### [Risk] 响应式布局在不同屏幕尺寸下的显示异常

**缓解措施：**
- 在 Chrome DevTools 中测试所有断点（xs/sm/md/lg/xl）
- 在真实设备上测试（iPhone SE, iPad Pro, 1920px 桌面）
- 使用 Element Plus 的响应式组件，避免手写媒体查询
- 为关键卡片添加最小宽度和溢出处理

### [Trade-off] 精简版流量图表信息量减少

**权衡：** 移动端精简版图表高度仅 60-80px，无法显示完整的历史趋势和网格线。

**缓解措施：**
- 点击图表可跳转到完整仪表盘页面查看详情
- 使用更明显的颜色区分上行/下行（橙色/蓝色）
- 保持曲线平滑和动画流畅，确保趋势可读

### [Trade-off] 导航栏流量指示器占用空间

**权衡：** 移动端 Header 空间有限，流量指示器可能挤压标题和用户菜单。

**缓解措施：**
- 使用紧凑型布局（图标 + 缩写数值，如 "↑2.5M"）
- 在超小屏幕（<360px）隐藏用户名，只显示头像图标
- 使用 el-dropdown 实现用户菜单，不占用横向空间

## Migration Plan

### 部署步骤

1. 创建新的 Vue 组件文件（在 web/src/components/dashboard/ 目录）
2. 修改 web/src/pages/Dashboard.vue，集成新组件并实现响应式布局
3. 修改 web/src/components/Layout/MainLayout.vue，在移动端 Header 添加 TrafficIndicator
4. 创建新的 composables（如 useTrafficGraph）管理流量图表状态
5. 运行 `npm run build` 确保编译无错误
6. 本地测试所有断点和交互功能
7. 提交代码并部署

### 回滚策略

- 使用 Git 分支管理，保持主分支稳定
- 如果出现严重问题，可快速回退到上一个稳定版本
- 新增组件不影响现有功能，可选择性禁用（通过条件渲染）

### 数据兼容性

- 无数据库迁移需求（仅前端变更）
- 无 API 破坏性变更（复用现有 WebSocket 和 REST API）
- 用户本地配置（如侧边栏折叠状态）不受影响

## Open Questions

1. **流量图表样式切换需求？**
   - 当前计划支持两种显示模式：折线图和面积图
   - 是否需要支持更多样式（如柱状图、填充渐变）？

2. **移动端精简版流量图表的交互方式？**
   - 点击跳转到完整仪表盘页面？
   - 点击显示详细信息弹窗？
   - 点击切换显示更多/更少数据点？

3. **统计卡片的颜色方案是否可自定义？**
   - 当前计划：上行橙色（secondary）、下行蓝色（primary）、连接绿色（success）、内存红色（error）
   - 是否需要支持主题切换（暗色模式）？

4. **导航栏流量指示器的详细交互？**
   - 是否需要悬停显示历史流量峰值？
   - 是否需要点击展开显示连接数、内存使用等额外信息？

5. **性能监控指标？**
   - 是否需要收集流量图表的渲染性能数据（FPS、绘制时间）？
   - 是否需要在性能不佳时自动降级（降低刷新率、简化曲线）？
