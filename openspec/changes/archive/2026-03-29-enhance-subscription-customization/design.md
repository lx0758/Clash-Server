## Context

当前订阅系统仅支持规则自定义，通过独立的 `Rule` 和 `Script` 表实现。用户需要更完整的自定义能力，包括代理、代理组、全局配置的调整。

现有架构：
- `Rule` 表：存储规则的 type、payload、proxy、mode、priority
- `Script` 表：存储后处理脚本
- `MergerService`：负责合并订阅内容和自定义配置

约束：
- 与 Subscription 是 1:1 关系
- YAML 格式配置，便于用户直接编辑
- 配置验证失败时需要回滚并提示错误

## Goals / Non-Goals

**Goals:**
- 统一的自定义配置存储模型
- 支持代理、代理组、规则的插入/追加/移除操作
- 支持全局配置覆盖
- 支持 YAML 格式输入
- 简化前端交互
- 完整的错误处理和验证机制

**Non-Goals:**
- proxy-providers / rule-providers 支持（与订阅功能重叠）
- 配置预览功能
- 多条自定义项管理（YAML 本身支持多项）
- 折叠交互（已取消）

## Decisions

### D1: 数据模型 - 单表存储

**决策**: 创建 `SubscriptionCustomization` 表，包含所有自定义字段。

**备选方案**:
1. 每种类型独立表 - 字段更清晰，但表多管理复杂
2. Subscription 表扩展字段 - 不符合职责分离原则
3. 单 JSON 字段 - SQL 查询不便

**选择理由**: 一个订阅只有一份自定义配置，单表 1:1 关系最简单直接。

### D2: 字段存储 - 多字段 vs JSON

**决策**: 使用多字段存储（proxy_insert, proxy_append, ...）。

**备选方案**:
- 单个 JSON 字段存储所有配置

**选择理由**: 字段语义明确，便于校验和扩展。

### D3: 前端交互 - 单弹窗 + Tab

**决策**: 一个"自定义配置"弹窗，内部 Tab 切换（代理/代理组/规则/全局/脚本）。

**备选方案**:
- 多个独立弹窗入口
- 折叠式输入区域

**选择理由**: 配置项关联性强，统一入口更便于管理；取消折叠简化交互。

### D4: 合并顺序

**决策**: 移除 → 插入 → 追加 → 全局覆盖 → 脚本后处理

**理由**: 先移除腾出空间，再插入/追加添加内容，最后覆盖全局配置，脚本做最终处理。

### D5: 脚本执行

**决策**: 支持两种写法：
1. 直接返回对象
2. 定义 `main(config)` 函数

**理由**: 兼容用户习惯，`main` 函数更符合编程直觉。

### D6: 验证与错误处理

**决策**: 多层验证 + 回滚机制：
1. 前端 YAML 格式验证
2. 后端 YAML 格式验证
3. 合并配置结构验证（proxies/proxy-groups/rules）
4. 核心应用验证
5. 任一步骤失败时恢复旧配置

**理由**: 防止无效配置破坏系统，提供清晰错误提示。

**错误消息**: 验证错误使用名称而非数组下标，便于用户理解。使用反射提取各种 map 类型的 `name` 字段。

### D7: 对话框交互

**决策**: 查看源文件、自定义配置、查看合并配置对话框：
- 点击外部不关闭（`close-on-click-modal="false"`）
- 操作按钮使用小尺寸（`size="small"`）

**理由**: 防止用户误操作关闭对话框丢失编辑内容；小按钮更紧凑。

**按钮文案**: "复制 YAML" → "复制"，"下载文件" → "下载"

## Risks / Trade-offs

**R1: YAML 格式校验** → 前端保存前校验 + 后端解析失败返回错误信息

**R2: 移除操作误删** → 用户对最终配置负责，不做额外限制

**R3: 原有数据迁移** → 当前无正式使用数据，直接删除旧表

**R4: 脚本执行错误** → 返回详细错误信息，不关闭弹窗

**R5: 核心应用失败** → 自动回滚到旧配置，显示错误提示

## Migration Plan

1. 创建新表 `subscription_customizations`
2. 删除 `rules`、`scripts` 表
3. 更新代码（模型、repository、service、handler）
4. 更新前端组件
5. 安装 `js-yaml` 和 `@types/js-yaml` 依赖
