## Context

当前前端存在两个独立页面：
- `Rules.vue` - 管理数据库中的自定义规则（insert/append 到配置）
- `Scripts.vue` - 管理数据库中的扩展脚本

这两个功能都是为订阅服务的配置扩展：
- 自定义规则：在订阅规则基础上插入或追加规则
- 扩展脚本：改写订阅生成的配置

用户需要查看 Clash Core 实时生效的规则列表，但当前无法实现。

**Core Rules API 数据结构**：
```json
{
  "rules": [
    {
      "index": 0,
      "type": "Domain",
      "payload": "example.com",
      "proxy": "DIRECT",
      "size": -1,
      "extra": {
        "disabled": false,
        "hitCount": 0,
        "hitAt": "...",
        "missCount": 0,
        "missAt": "..."
      }
    }
  ]
}
```

数据量较大（可能数千条），需要虚拟滚动优化。

## Goals / Non-Goals

**Goals:**
- 新"规则"页面展示 Core 实时规则列表（只读）
- "订阅"页面新增"规则"和"脚本"两个 Tab，整合自定义配置
- 后端新增 Core rules API 代理
- 简化导航，移除独立的"脚本"页面

**Non-Goals:**
- 不修改规则匹配逻辑
- 不改变数据库中规则/脚本的数据结构
- 不支持规则编辑（Core 规则是只读的）

## Decisions

### 1. 订阅页面布局

**选择**：Tab 切换布局

```
┌─────────────────────────────────────────────────┐
│  [订阅] [规则] [脚本]                            │
├─────────────────────────────────────────────────┤
│                                                 │
│              当前 Tab 内容                       │
│                                                 │
└─────────────────────────────────────────────────┘
```

**理由**：
- 规则和脚本都是订阅的配置扩展，放一起逻辑清晰
- 减少导航层级，用户在订阅页面即可完成所有配置
- Tab 切换比分栏更适合这三个功能的关系（平级而非并排对比）

### 2. Core 规则展示方案

**选择**：前端直接调用后端代理 API，后端转发到 Core

**理由**：
- 与现有架构一致（proxies、connections 都采用此模式）
- 便于统一认证和错误处理

### 3. 大数据量处理

**选择**：使用 CSS 虚拟滚动（VueUse 的 `useVirtualList`）

**理由**：
- 规则数量可能达数千条
- 避免页面卡顿
- 纯 CSS 实现，无需引入重型库

## Risks / Trade-offs

**[规则数量大导致性能问题]**
- 缓解：使用虚拟滚动，仅渲染可视区域

**[Core 未运行时无数据]**
- 缓解：显示"核心未运行"提示，与连接页面行为一致

**[订阅页面功能过多]**
- 缓解：Tab 布局保持清晰的视觉分隔
