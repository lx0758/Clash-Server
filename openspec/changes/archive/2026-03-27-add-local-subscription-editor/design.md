## Context

后端已有 `PUT /api/subscriptions/:id/content` API 用于更新订阅内容，但缺少获取内容的 API。前端没有查看/编辑订阅内容的界面。本地订阅没有远程 URL，用户需要直接编辑 `content` 字段；远程订阅用户也希望查看订阅内容。此外，本地订阅在列表中缺少节点数量和编辑时间等信息展示。

## Goals / Non-Goals

**Goals:**
- 为所有订阅提供内容查看界面
- 为本地订阅提供内容编辑功能
- 远程订阅内容只读
- 优化本地订阅的列表展示
- 支持复制和下载功能
- 优化小屏幕显示效果

**Non-Goals:**
- 不支持远程订阅的内容编辑（远程订阅内容来自 URL 刷新）
- 不提供 YAML 语法高亮（可后续增强）

## Decisions

### 1. 使用 textarea 作为编辑器

**决定**: 使用 textarea 编辑 YAML 内容，设置 `white-space: pre` 禁止自动折行

**理由**:
- 项目未集成代码编辑器库（如 Monaco、CodeMirror）
- 避免引入新依赖
- 功能足够满足基本编辑需求

**备选**: 集成 Monaco Editor
- 优点：语法高亮、自动补全
- 缺点：增加包体积、需要额外配置

### 2. 统一入口，区分行为

**决定**: 所有订阅显示"查看源文件"，本地订阅可编辑，远程订阅只读

**理由**:
- UI 一致性更好
- 用户能查看所有订阅内容
- 通过 readonly 属性区分行为
- 命名与"查看合并配置"区分开

### 3. 新增获取内容 API

**决定**: 新增 `GET /api/subscriptions/:id/content` 返回原始内容

**理由**:
- 现有 GET /api/subscriptions/:id 返回完整订阅信息，但 content 字段可能很大
- 独立 API 便于按需加载
- 减少列表接口的数据量

### 4. 本地订阅展示优化

**决定**: 在收缩状态下显示节点数量和编辑时间

**理由**:
- 与远程订阅展示一致
- 用户能快速了解订阅状态

### 5. 小屏幕优化

**决定**: 参考 MergedConfigDialog 的 fullscreen 样式

**理由**:
- 保持 UI 一致性
- 充分利用屏幕空间

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| YAML 格式错误导致解析失败 | 提交前验证基本格式，显示错误提示 |
| 内容过大导致编辑器卡顿 | textarea 对几千行配置足够流畅 |
| 获取内容 API 权限问题 | 复用现有认证中间件 |
